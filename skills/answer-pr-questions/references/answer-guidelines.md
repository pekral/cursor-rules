# Answer Guidelines

## Audience

All answers must be written for **non-technical stakeholders** — project managers, clients, or business owners. They should not need programming knowledge to understand the response.

## Language rules

- No technical jargon (no function names, class names, file paths, or code references)
- Clear business meaning — explain impact, not implementation
- Maximum 2-4 short sentences per answer
- Use plain, direct language

## Handling incomplete information

- If a question cannot be fully answered from available information, explicitly state:
  - What part of the answer is known
  - What information is still missing
  - Who should provide the missing information (developer, PM, client, etc.)

## Answer quality checklist

- [ ] Would a non-programmer understand this answer without further explanation?
- [ ] Does the answer address the actual question, not a related topic?
- [ ] Is the answer concise (2-4 sentences maximum)?
- [ ] If information is missing, is the gap clearly identified?
- [ ] Is the language consistent (single language, no mixing)?

## Unified answer principle

When multiple comments or threads discuss the same topic, produce **one unified final answer** that synthesizes all available information — do not repeat each comment separately.
