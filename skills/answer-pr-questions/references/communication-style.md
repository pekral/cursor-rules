# Communication Style

## Professional Tone
- Write as a senior developer speaking to a peer.
- Be direct — state the answer first, then explain.
- Stay neutral and factual — avoid emotional language.

## Concise Communication
- One idea per sentence.
- Remove filler words ("basically", "actually", "just", "I think").
- If the answer is one sentence, do not pad it.

## Respectful Disagreement
- Acknowledge the reviewer's perspective before presenting yours.
- Use evidence (code behavior, specs, constraints) — not opinion.
- Frame as trade-off discussion, not right vs wrong.
- Example: "That approach would work, but it adds a dependency on X which we avoid because..."

## Avoiding Passive-Aggressive Tone
- Never use "as I already mentioned" or "as stated above".
- Never imply the reviewer should have known something.
- If a question was already answered elsewhere, link to the answer without commentary.

## Choosing Response Length
- Trivial question → one sentence.
- Design decision → 2-4 sentences with reasoning.
- Complex trade-off → short paragraph with code reference.
- Never exceed what the question requires.
