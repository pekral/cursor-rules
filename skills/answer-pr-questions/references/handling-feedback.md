# Handling Feedback

## How to Handle Criticism
- Separate the feedback from the tone — focus on the technical content.
- If the criticism is valid, acknowledge it directly: "Good catch, fixing this."
- If the criticism is partially valid, acknowledge the valid part and clarify the rest.
- Never dismiss feedback without evidence.

## When to Accept
- The suggestion improves readability, correctness, or maintainability.
- The suggestion aligns with project conventions.
- The cost of the change is low and the benefit is clear.
- Accept quickly and move on — do not over-explain agreement.

## When to Push Back
- The suggestion would break existing behavior or introduce regression.
- The suggestion conflicts with project constraints or architecture decisions.
- The suggestion adds unnecessary complexity for marginal benefit.
- Always provide evidence when pushing back — never "I prefer it this way."

## How to Acknowledge Mistakes
- State the mistake clearly: "You are right, this misses the null case."
- Describe the fix briefly: "Added a guard clause at line 42."
- Do not over-apologize or add unnecessary context.
- Move forward — one sentence is enough.

## Handling Repeated Questions
- If the same question comes up again, the original answer was unclear.
- Rephrase the answer — do not repeat it verbatim.
- Consider adding a code comment if the confusion is likely to recur.
