# Summary Guidelines

## Tone and Audience

- Write the summary in **singular first person** (one developer made the changes).
- The summary must be understandable by both **developers** and **product managers**.
- Focus on the **"why"** and **business impact**, not on implementation details.

## Content Rules

- Do NOT include code snippets unless they are essential to explain a breaking change.
- Group changes into logical categories (e.g., new features, bug fixes, refactoring, configuration, tests).
- If no breaking changes exist, explicitly state "No breaking changes."

## Formatting

- Output must be formatted in **markdown**.
- Use the section structure defined in the output contract.
- Keep the "What changed" section concise (3-5 sentences).
